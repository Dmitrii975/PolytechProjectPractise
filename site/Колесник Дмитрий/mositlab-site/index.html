Проблема
Почему это важно?

Без системы приоритизации компания теряет огромные ресурсы на заявки, которые никогда не станут сделками.

85%

рабочего времени ведущих специалистов уходит на обработку заявок, которые никогда не конвертируются в продажи. Каждая заявка требует полной конструкторско-технологической проработки.

Только 10–15% из всех входящих заявок переходят в реальные сделки. Без приоритизации специалисты не знают, какие заказы обрабатывать в первую очередь.

Аналитика, Трассировка
Этап 01
Анализ и сбор данных
Глубокий анализ бизнес-процессов «as-is» в МОСИТЛАБ. Получены реальные исторические данные о заказах для обучения модели. Разработана схема предварительной обработки и обогащения заявок. Проведено более 15 интервью с ключевыми специалистами компании.
Этап 02
Разработка AI-модели ранжирования
Создан и протестирован MVP AI-модели на базе градиентного бустинга. Модель оценивает перспективность каждой заявки по десяткам факторов: маржинальность, вероятность закрытия, сложность исполнения, история клиента. Результат — приоритет High / Medium / Low с объяснимостью решений.
Этап 03
Backend и интеграционная архитектура
Реализована серверная часть, REST API, механизмы автоматической загрузки заявок из существующих систем заказчика (CRM, 1C ERP) и расчёта приоритетов в реальном времени. Система работает в асинхронном режиме с гарантией доставки результатов.
Этап 04
Веб-интерфейс и тестирование
Создан удобный дашборд для работы с приоритетами, реализована полная трассировка решений. Проведено end-to-end тестирование на реальных данных. Симуляция Монте-Карло подтвердила значительный рост маржинального дохода — до 3× по оптимистичному сценарию.

История разработки

Ключевые события и достижения проекта в хронологическом порядке.

04 · ЭТАП IVФинальный

Запущен веб-интерфейс и проведено комплексное тестирование

Создан удобный дашборд для работы с приоритетами заявок. Интерфейс отображает распределение по уровням приоритета, детальную карточку каждой заявки с объяснением решения AI, тренды и аналитику. Реализована полная трассировка — менеджер видит, какие факторы повлияли на оценку. Проведено end-to-end тестирование на реальных данных МОСИТЛАБ. Симуляция Монте-Карло (10 000 итераций) подтвердила экономический эффект: рост маржинального дохода в 1.7–3.0× в зависимости от сценария. Система готова к пилотному внедрению.

✓ Готово к внедрению · Монте-Карло · 10 000 итераций
03 · ЭТАП IIIBackend

Разработан backend и интеграционная архитектура

Реализована серверная часть на FastAPI (Python). Спроектированы и задокументированы REST API endpoints для управления заявками, получения приоритетов и доступа к аналитике. Настроена очередь задач на Celery + Redis для асинхронного расчёта приоритетов. Разработаны коннекторы для автоматической загрузки заявок из CRM и 1C ERP МОСИТЛАБ. Интеграция работает в режиме реального времени с polling-интервалом 5 минут. Вся инфраструктура упакована в Docker Compose для простого развёртывания.

FastAPI · Celery · Docker
02 · ЭТАП IIMachine Learning

Внедрён алгоритм ранжирования на основе машинного обучения

Создан и протестирован MVP AI-модели, которая оценивает перспективность каждой заявки по десяткам факторов и присваивает приоритет (HIGH / MEDIUM / LOW). Испытано несколько алгоритмов: Logistic Regression, Random Forest, XGBoost. По совокупности метрик (F1-score, ROC-AUC, скорость инференса) выбран XGBoost. Достигнута точность 81% на отложенной выборке. Внедрена объяснимость решений через SHAP-значения — специалист видит, почему заявке присвоен тот или иной приоритет.

XGBoost · F1=0.81 · AUC=0.87
01 · ЭТАП IАнализ

Добавлен модуль анализа и сбора данных

Команда завершила глубокий анализ бизнес-процессов «as-is» в МОСИТЛАБ и получила реальные исторические данные о заказах для обучения модели. Проведено более 15 интервью с ключевыми специалистами компании — коммерческим директором, руководителями отделов и инженерами-конструкторами. Составлен реестр всех факторов, влияющих на успешность сделки. Разработана схема предварительной обработки и обогащения заявок: нормализация, обработка пропусков, кодирование категориальных переменных. Результаты задокументированы в виде Data Specification документа.
